# slipstream-trackdata

Public, openly-licensed **track imagery** for the Slipstream Live iRacing coaching overlay — served
**on demand**. The app downloads each venue's satellite imagery the first time you drive there and caches it
locally (`%APPDATA%\SlipstreamLive\satellite-cache\`), so the installer stays small and the track list can grow
without an app update.

This repo is **public on purpose**: the app fetches assets anonymously at runtime, and the imagery is derived
from public-domain / open-licensed ortho sources (it is *not* proprietary — the Slipstream Live app itself is a
separate, private repo). See [`ATTRIBUTION.md`](ATTRIBUTION.md) for the per-source credits the licenses require.

## How it's distributed

- **Big binaries live in GitHub Releases, not in git history.** Each venue is one zip asset
  (`{venue-slug}.zip`) attached to a release (e.g. tag `imagery-v1`). The `satellite/` working tree and
  `dist/` are **gitignored** — committing ~hundreds of MB of JPEGs would bloat the repo forever.
- **`index.json`** (committed, small) is the manifest the app reads: it maps each iRacing track id → its venue,
  the release asset to download, and a version. Bump a track's version when you regenerate it → the app
  re-downloads just that one.

```
index.json            # ir_id -> { venue, assetUrl, version, corners, attribution }   (committed)
ATTRIBUTION.md        # per-source credits (generated, committed)
tools/package_release.py   # build per-venue zips + index.json + ATTRIBUTION from a satellite/ tree
satellite/            # imagery working copy (GITIGNORED — regenerable; distributed via Releases)
dist/                 # built zips (GITIGNORED)
```

## Cutting a release

The imagery is generated by the (private) SlipstreamTrackInsights pipeline. To publish a new set:

```bash
# 1. build per-venue zips + index.json + ATTRIBUTION.md from a satellite/ source tree
python tools/package_release.py --src ../SlipstreamLive/trackdata/satellite --out dist --version imagery-v1

# 2. commit the regenerated metadata
git add index.json ATTRIBUTION.md && git commit -m "imagery-v1" && git push

# 3. attach the zips as release assets (gh CLI, or the web UI)
gh release create imagery-v1 dist/*.zip --title "Imagery v1" --notes "..."
```

The app reads `index.json` (raw URL or bundled copy) to discover versions, then pulls each venue's zip on demand.

### Updating one track without making every client re-download

Build a partial release and give it a unique version tag:

```bash
python tools/package_release.py --src ../SlipstreamLive/trackdata/satellite \
  --meta ../SlipstreamLive/trackdata/iracing-tracks-metadata.json \
  --out dist --version imagery-roadamerica-20260714 --only-ids 18
```

`--only-ids` preserves every other entry in the existing `index.json`, builds only the selected venue zip,
and changes only those track entries to the new version/asset URL/hash. After that zip is attached to the
matching GitHub release and the updated index is pushed, installed clients keep serving their cached image,
download that one changed zip in the background, then atomically replace and hot-reload it.
